What is Git?

Git is a distributed version control system (DVCS) created by Linus Torvalds in 2005 for managing the Linux kernel source code. It tracks changes in files over time and allows multiple people to collaborate without overwriting each other's work.

🕒

Version Control

Records every change ever made to files. Travel back to any point in history instantly.

📦

Distributed

Every developer has a full copy of the entire project history — no single point of failure.

🌿

Branching

Create isolated branches to experiment and develop features without touching the main code.

⚡

Speed

Nearly all operations are local — blazing fast because no network calls needed for history.

Git vs GitHub

GitGitHub
Local tool installed on your machineCloud hosting platform for Git repositories
Command-line programWeb interface + collaboration features
Manages version history locallyAdds PRs, Issues, Actions, Wiki, etc.
Free, open sourceFree tier + paid plans (owned by Microsoft)
💡 Key insight GitHub, GitLab, and Bitbucket are all "remotes" — they host your Git repositories online. Git is the tool; GitHub is one place to store and share that tool's output.

Git Object Model

Internally, Git stores everything as 4 types of objects, each identified by a SHA-1 hash:

  COMMIT OBJECT                 TREE OBJECT               BLOB OBJECT
  ┌─────────────────┐           ┌──────────────┐          ┌──────────┐
  │ tree: a1b2c3    │──────────►│ README.md    │─────────►│ # Hello  │
  │ parent: f9e8d7  │           │   blob: 7f8a │          │ World    │
  │ author: Priya   │           │ src/         │          └──────────┘
  │ date: 2024-01   │           │   tree: 2b3c │
  │ message: "init" │           └──────────────┘
  └─────────────────┘                  │
                                       ▼
                              ┌──────────────────┐
                              │ TAG OBJECT       │
                              │ (annotated tags) │
                              │ tag: v1.0        │
                              │ tagger: Priya    │
                              └──────────────────┘
🔑 Mental Model Think of Git as a content-addressable filesystem. Files are stored by their SHA-1 hash. If two files have identical content, Git stores only one copy. Every commit is a complete snapshot, not a diff.

The .git Directory

  .git/
  ├── HEAD          ← Points to current branch (ref: refs/heads/main)
  ├── config        ← Local repo configuration
  ├── index         ← Staging area (binary file)
  ├── objects/      ← All Git objects (blobs, trees, commits, tags)
  │   ├── info/
  │   └── pack/
  ├── refs/
  │   ├── heads/    ← Local branch pointers
  │   │   ├── main
  │   │   └── feature/login
  │   └── tags/     ← Tag pointers
  │       └── v1.0
  ├── logs/         ← Reflog history
  └── hooks/        ← Scripts that run on Git events

bash

Stashing

Temporarily save work without committing:

bash

Branches

A branch is simply a lightweight movable pointer to a commit. Creating branches in Git is nearly instantaneous — it just creates a new 41-byte file in .git/refs/heads/.

main ●────●────●────────────────● ← HEAD │ ↑ feature └────●────●───╯ │ hotfix └────● v1.1
bash
# ── Creating branches ──────────────────────────────────── git branch feature/login # Create branch (stay on current) git checkout -b feature/login # Create + switch (old syntax) git switch -c feature/login # Create + switch (modern, preferred) git switch -c hotfix origin/main # Create from remote branch # ── Switching branches ─────────────────────────────────── git checkout main # Old syntax git switch main # Modern syntax git switch - # Switch to previous branch # ── Listing branches ───────────────────────────────────── git branch # Local branches git branch -r # Remote tracking branches git branch -a # All branches git branch -v # With last commit info git branch --merged # Branches merged into current git branch --no-merged # Branches NOT merged yet # ── Renaming branches ──────────────────────────────────── git branch -m old-name new-name # Rename local branch git branch -m new-name # Rename current branch # ── Deleting branches ──────────────────────────────────── git branch -d feature/login # Delete (safe - merged only) git branch -D feature/login # Force delete (even unmerged) git push origin --delete feature/login # Delete remote branch # ── Tags ───────────────────────────────────────────────── git tag v1.0 # Lightweight tag git tag -a v1.0 -m "Release 1.0" # Annotated tag (preferred) git tag -a v1.0 abc123 # Tag specific commit git tag # List all tags git push origin v1.0 # Push tag to remote git push origin --tags # Push all tags git tag -d v1.0 # Delete local tag git push origin --delete v1.0 # Delete remote tag

Merging & Rebasing

Merge Types

FAST-FORWARD MERGE                     3-WAY MERGE (no-ff)

  main: A─B                             main: A─B─────────E
                │ merge feature          feature: A─B─C─D─╯
  feature: A─B─C─D                       (creates merge commit E)


SQUASH MERGE                           REBASE
  feature: A─B─C─D    squash            feature: A─B─C─D
                 │                             rebase main ──►  A─B─E─C'─D'
  main: A─B──────S                                           (replays on top)
    (S contains all changes in one commit)
bash
# ── Merge ──────────────────────────────────────────────── git switch main git merge feature/login # Merge (fast-forward if possible) git merge --no-ff feature/login # Always create merge commit git merge --squash feature/login # Squash all commits into one git merge --abort # Abort in-progress merge # ── Rebase ─────────────────────────────────────────────── git switch feature/login git rebase main # Replay feature commits on top of main git rebase --abort # Abort rebase git rebase --continue # Continue after resolving conflict git rebase --skip # Skip conflicting commit # ── Interactive Rebase (powerful!) ─────────────────────── git rebase -i HEAD~3 # Rewrite last 3 commits interactively git rebase -i abc123 # Rewrite commits after abc123 # In the editor you can: pick, squash, fixup, drop, reword, edit # ── Cherry-pick ────────────────────────────────────────── git cherry-pick abc123 # Copy a commit to current branch git cherry-pick abc123..def456 # Cherry-pick a range git cherry-pick --no-commit abc123 # Apply changes without committing
⚠️ Golden Rule of Rebasing Never rebase commits that have been pushed to a shared remote branch. Rebasing rewrites commit history; doing so on public branches causes chaos for teammates.

Merge vs Rebase Comparison

MergeRebase
Preserves full history (non-destructive)Creates linear history (cleaner log)
Creates merge commitsNo merge commits
Safe for public branchesOnly use on private/local branches
Shows exactly what happenedHistory looks like straight line
git merge featuregit rebase main

Merge Conflicts — Solving Step by Step

A merge conflict occurs when two branches modify the same line(s) of the same file, or one branch deletes a file while another modifies it. Git cannot auto-decide which change to keep — you must resolve it manually.

When Do Conflicts Happen?

  main branch:     --- A --- B(changes line 5) ---
                             \                     \
  feature branch:             C(changes line 5)    MERGE → CONFLICT!

Conflict Markers

Git marks the conflicting areas in the file like this:

<<<<<<< HEAD
const greeting = "Hello World";
const color = "blue";
======= (separator)
const greeting = "Hi There!";
const color = "red";
>>>>>>> feature/update-ui
  • <<<<<<< HEAD — your current branch's version
  • ======= — separator between versions
  • >>>>>>> feature/update-ui — incoming branch's version

Complete Conflict Resolution Workflow

bash
# ── Step 1: Attempt merge ──────────────────────────────── git switch main git merge feature/update-ui # Output: CONFLICT (content): Merge conflict in src/app.js # Automatic merge failed; fix conflicts and commit # ── Step 2: See what's conflicting ─────────────────────── git status # both modified: src/app.js git diff # See all conflicts # ── Step 3: Open and edit the file ─────────────────────── # Manually edit the file — decide which code to keep: # Option A: Keep ours → delete <<<, ===, >>> lines + their code # Option B: Keep theirs → delete <<<, ===, >>> lines + our code # Option C: Keep both → merge the code manually, remove markers # ── Step 4: Mark as resolved ───────────────────────────── git add src/app.js # Mark file as resolved # ── Step 5: Complete the merge ─────────────────────────── git commit # Opens editor with merge commit message # Or: git commit -m "merge: resolve conflict in app.js" # ── OR: Abort if needed ────────────────────────────────── git merge --abort # Return to pre-merge state

Conflict Resolution Tools

bash
# ── Built-in merge tool ────────────────────────────────── git mergetool # Opens configured merge tool # ── Configure VS Code as merge tool ────────────────────── git config --global merge.tool vscode git config --global mergetool.vscode.cmd 'code --wait $MERGED' # ── Shortcuts during conflict ───────────────────────────── git checkout --ours src/app.js # Take our version entirely git checkout --theirs src/app.js # Take their version entirely # ── Rebase conflict resolution ─────────────────────────── git rebase main # (resolve conflicts in editor) git add . git rebase --continue # Continue to next commit # Repeat until rebase completes
🔧 Pro Tips Use VS Code's built-in merge editor (3-panel view) for complex conflicts. Click "Accept Current", "Accept Incoming", or "Accept Both" buttons instead of editing markers manually.

GitHub & Remotes

  ┌────────────────────────────────────────────────────────────────┐
  │                         GITHUB.COM                            │
  │                                                                │
  │  ┌──────────────────┐        ┌──────────────────────────┐     │
  │  │  UPSTREAM REPO   │        │    ORIGIN (your fork)    │     │
  │  │  original/repo   │        │    yourname/repo          │     │
  │  └──────────────────┘        └──────────────────────────┘     │
  │           │  PR & Review               │ push/pull             │
  └───────────┼────────────────────────────┼───────────────────────┘
              │ git fetch upstream          │
              │                            │ git push/pull origin
              ▼                            ▼
  ┌───────────────────────────────────────────────────────────────┐
  │                     LOCAL MACHINE                             │
  │  origin/main (remote tracking)  →  main (local)              │
  │  upstream/main (remote tracking) →  (merged via PR)          │
  └───────────────────────────────────────────────────────────────┘
bash
# ── Managing remotes ───────────────────────────────────── git remote -v # List remotes with URLs git remote add origin git@github.com:user/repo.git git remote add upstream git@github.com:original/repo.git git remote remove upstream # Remove remote git remote rename origin new-name # Rename remote git remote set-url origin git@github.com:user/new-repo.git # Change URL # ── Fetching ───────────────────────────────────────────── git fetch origin # Download all remote changes (no merge) git fetch origin main # Fetch specific branch git fetch --all # Fetch from all remotes git fetch --prune # Remove deleted remote branches # ── Pulling ────────────────────────────────────────────── git pull # fetch + merge (current branch) git pull origin main # Pull specific branch git pull --rebase # fetch + rebase (cleaner history) git pull --ff-only # Only pull if fast-forward possible # ── Pushing ────────────────────────────────────────────── git push origin main # Push to remote git push -u origin feature/login # Push + set upstream tracking git push --force-with-lease # Safer force push (checks for new commits) git push --force # Force push (DANGEROUS — overwrites remote) git push origin --delete feature # Delete remote branch git push --all # Push all local branches

Tracking Branches

bash
# Remote tracking = local refs to last-known state of remote branches git branch -vv # Show tracking info for all branches # Output example: # main a1b2c3d [origin/main] fix: update header # feat e5f6a7b [origin/feat: ahead 2] add feature git branch --set-upstream-to=origin/main main # Set tracking manually

Collaboration Workflow

Fork & Pull Request Workflow (Open Source)

  STEP 1: Fork        STEP 2: Clone        STEP 3: Branch
  ┌───────────┐       ┌──────────────┐     ┌──────────────────┐
  │ original/ │ Fork► │  yourname/   │ ──► │  feature/login   │
  │   repo    │       │    repo      │     └──────────────────┘
  └───────────┘       └──────────────┘              │
         ▲                                           │
         │  Pull Request (Merge)      STEP 4: Code   │
         └─────────────────────────── Push & PR ◄───┘
bash
# ── Step 1: Fork on GitHub UI, then clone your fork ────── git clone git@github.com:yourname/repo.git cd repo # ── Step 2: Add upstream ───────────────────────────────── git remote add upstream git@github.com:original/repo.git git remote -v # should show both origin and upstream # ── Step 3: Create feature branch ──────────────────────── git switch -c feature/my-feature # ── Step 4: Make changes, commit ───────────────────────── git add . git commit -m "feat: add awesome feature" # ── Step 5: Sync with upstream before pushing ──────────── git fetch upstream git rebase upstream/main # Rebase on top of latest main # ── Step 6: Push and open PR ───────────────────────────── git push -u origin feature/my-feature # Open PR on GitHub: yourname/repo → original/repo # ── Step 7: After PR merged, clean up ──────────────────── git switch main git pull upstream main git push origin main git branch -d feature/my-feature git push origin --delete feature/my-feature

Team Repository Workflow (Company/Internal)

bash
# Clone shared repo (no fork needed) git clone git@github.com:company/repo.git # Work on feature branch git switch -c feature/TICKET-123-auth # Push to remote git push -u origin feature/TICKET-123-auth # Keep up to date with team changes git fetch origin git rebase origin/main # Rebase on latest main # Or merge (if team prefers) git merge origin/main

Pull Requests & Code Review

A Pull Request (PR) is a GitHub feature — not a Git feature — to propose, discuss, review, and merge changes.

PR Lifecycle

  Push Branch  →  Open PR  →  Review  →  Address Feedback  →  Approved  →  Merge
       │                                                                        │
       └── CI/CD checks run automatically ────────────────────────────────────►┘

GitHub CLI for PRs

bash
# Install: brew install gh or winget install GitHub.cli # ── Auth ──────────────────────────────────────────────── gh auth login # ── Create PR ─────────────────────────────────────────── gh pr create gh pr create --title "feat: add login" --body "Closes #123" gh pr create --draft # Draft PR (not ready for review) gh pr create --base develop # Target specific base branch # ── View PRs ──────────────────────────────────────────── gh pr list # List open PRs gh pr view 42 # View PR #42 gh pr checkout 42 # Check out PR #42 locally gh pr diff 42 # View diff of PR # ── Review ────────────────────────────────────────────── gh pr review 42 --approve gh pr review 42 --request-changes --body "please fix tests" gh pr review 42 --comment -b "LGTM mostly" # ── Merge ─────────────────────────────────────────────── gh pr merge 42 # Interactive merge gh pr merge 42 --squash # Squash and merge gh pr merge 42 --rebase # Rebase and merge gh pr merge 42 --delete-branch # Delete branch after merge

Branch Protection Rules

Configure in GitHub: Repo Settings → Branches → Branch protection rules

  • Require pull request reviews before merging (min 1 or 2 approvals)
  • Require status checks to pass (CI/CD must be green)
  • Require branches to be up to date before merging
  • Restrict who can push to matching branches
  • Require signed commits

Advanced Git Commands

Reset, Revert & Restore

  git reset --soft HEAD~1   → Undo commit, keep changes STAGED
  git reset --mixed HEAD~1  → Undo commit, keep changes in WORKING DIR (default)
  git reset --hard HEAD~1   → Undo commit, DISCARD changes (DESTRUCTIVE!)

  git revert abc123         → Create NEW commit that undoes abc123 (safe for shared)
  git restore file.txt      → Restore file to last committed state
bash
# ── Reset ──────────────────────────────────────────────── git reset --soft HEAD~1 # Undo last commit, staged changes git reset --mixed HEAD~1 # Undo last commit, unstaged changes git reset --hard HEAD~1 # Undo last commit, discard all changes git reset --hard abc123 # Reset to specific commit # ── Revert (safe — for shared branches) ────────────────── git revert abc123 # Create undo commit for abc123 git revert HEAD~3..HEAD # Revert last 3 commits git revert --no-commit abc123 # Stage revert without committing # ── Reflog (your safety net) ────────────────────────────── git reflog # Full history of HEAD movements git reflog show feature/login # Reflog for specific branch git reset --hard HEAD@{3} # Recover to a specific reflog entry # ── Bisect (find buggy commit) ─────────────────────────── git bisect start git bisect bad # Current commit is bad git bisect good v1.0 # v1.0 tag was good # Git checks out middle commit → test it → mark good/bad git bisect good # or: git bisect bad git bisect reset # End bisect session # ── Blame ──────────────────────────────────────────────── git blame file.txt # Show who last changed each line git blame -L 10,25 file.txt # Blame specific lines 10–25 # ── Worktree ───────────────────────────────────────────── git worktree add ../hotfix hotfix # Work on two branches simultaneously git worktree list git worktree remove ../hotfix # ── Submodules ─────────────────────────────────────────── git submodule add https://github.com/lib/lib.git libs/lib git submodule update --init --recursive git submodule foreach git pull origin main

GitHub Actions

GitHub Actions is a CI/CD and automation platform built directly into GitHub. You define workflows in YAML files stored at .github/workflows/.

Architecture & Key Concepts

  ┌──────────────────────────────────────────────────────────────────┐
  │                     GITHUB ACTIONS ARCHITECTURE                  │
  │                                                                   │
  │  EVENT (trigger)           WORKFLOW (.github/workflows/*.yml)     │
  │  ─────────────────────     ──────────────────────────────────     │
  │  • push to branch          WORKFLOW                               │
  │  • pull_request            └── JOB 1 (runs-on: ubuntu-latest)    │
  │  • schedule (cron)              ├── STEP 1: checkout             │
  │  • workflow_dispatch            ├── STEP 2: setup-node           │
  │  • release                      ├── STEP 3: npm install          │
  │  • issue events                 └── STEP 4: npm test             │
  │                            └── JOB 2 (depends on JOB 1)         │
  │  RUNNER (execution env)         ├── STEP 1: build                │
  │  ─────────────────────          └── STEP 2: deploy               │
  │  • GitHub-hosted                                                  │
  │    ubuntu/windows/macos    JOBS run in PARALLEL by default       │
  │  • Self-hosted             STEPS run SEQUENTIALLY within a job   │
  └──────────────────────────────────────────────────────────────────┘
ConceptDescription
WorkflowAutomated process defined in a YAML file. Triggered by events.
EventTrigger that starts a workflow (push, PR, schedule, manual)
JobSet of steps that run on the same runner. Jobs run in parallel by default.
StepIndividual task: a shell command or a reusable Action.
ActionReusable unit of code from the Actions Marketplace.
RunnerServer that executes the workflow (GitHub-hosted or self-hosted).

Workflow 1: Basic CI (Test on every PR)

📄 .github/workflows/ci.yml 
name: CI

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest

    strategy:
      matrix:
        node-version: [ 18.x, 20.x ]  # Test on multiple Node versions

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js ${{ matrix.node-version }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: npm

      - name: Install dependencies
        run: npm ci

      - name: Run linter
        run: npm run lint

      - name: Run tests
        run: npm test -- --coverage

      - name: Upload coverage report
        uses: actions/upload-artifact@v4
        with:
          name: coverage-report
          path: coverage/

Workflow 2: Build & Deploy to Production

📄 .github/workflows/deploy.yml 
name: Deploy to Production

on:
  push:
    branches: [ main ]  # Only on merge to main

env:
  NODE_VERSION: '20.x'
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20.x', cache: npm }
      - run: npm ci && npm test && npm run build
      - uses: actions/upload-artifact@v4
        with: { name: build-output, path: dist/ }

  docker-build:
    needs: build-and-test   # Depends on previous job
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    steps:
      - uses: actions/checkout@v4
      - name: Log in to registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          push: true
          tags: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest

  deploy:
    needs: docker-build
    runs-on: ubuntu-latest
    environment: production  # Requires manual approval
    steps:
      - name: Deploy to server
        run: |
          echo "Deploying to production..."
          # SSH into server and pull new image
        env:
          DEPLOY_KEY: ${{ secrets.DEPLOY_SSH_KEY }}
          SERVER_HOST: ${{ secrets.SERVER_HOST }}

Workflow 3: Full Python CI/CD + PyPI Release

📄 .github/workflows/python-ci.yml 
name: Python CI/CD

on:
  push:
    branches: [ main ]
  release:
    types: [ published ]   # Trigger on GitHub Release

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ['3.10', '3.11', '3.12']
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
      - run: |
          pip install -r requirements-dev.txt
          pytest --cov=mypackage --cov-report=xml
      - uses: codecov/codecov-action@v4
        with:
          token: ${{ secrets.CODECOV_TOKEN }}

  publish:
    if: github.event_name == 'release'
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
      - run: |
          pip install build twine
          python -m build
          twine upload dist/*
        env:
          TWINE_USERNAME: __token__
          TWINE_PASSWORD: ${{ secrets.PYPI_API_TOKEN }}

Workflow 4: Scheduled Tasks & Notifications

📄 .github/workflows/scheduled.yml 
name: Scheduled Tasks

on:
  schedule:
    - cron: '0 9 * * 1-5'   # 9 AM every weekday (UTC)
  workflow_dispatch:            # Allow manual trigger from UI
    inputs:
      environment:
        description: 'Target environment'
        required: true
        default: staging
        type: choice
        options: [staging, production]

jobs:
  health-check:
    runs-on: ubuntu-latest
    steps:
      - name: Check API health
        run: |
          STATUS=$(curl -s -o /dev/null -w "%{http_code}" https://api.example.com/health)
          if [ $STATUS -ne 200 ]; then
              echo "Health check failed: $STATUS"
              exit 1
            fi
      - name: Notify Slack on failure
        if: failure()    # Only runs if previous step failed
        uses: slackapi/slack-github-action@v1.26.0
        with:
          payload: '{"text": "🚨 API health check failed!"}'
        env:
          SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK }}

📸 Stage & Commit

git stashStash changes

git show <hash>Show commit
🌿 Branches

🔄 Rebase & Reset

git reset --hard HEAD~1Discard last commit
git revert <hash>Safe undo

git push -u origin <br>Push + track

git bisect startBinary search bug

git push origin --tagsPush all tags

git shortlog -snCommit count by author
git log --statFiles changed per commit
📌 Daily Git Workflow Summary
  1. Pull latest: git pull --rebase origin main
  2. Create branch: git switch -c feature/my-feature
  3. Code & stage: git add -p
  4. Commit: git commit -m "feat: describe change"
  5. Push: git push -u origin feature/my-feature
  6. Open PR on GitHub → review → merge → delete branch
